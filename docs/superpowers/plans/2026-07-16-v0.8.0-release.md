# v0.8.0 Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish the accumulated post-v0.7.0 changes as an immutable, fully verified `v0.8.0` GitHub/GHCR/LXC release.

**Architecture:** Prepare release metadata on the existing `release/v0.8.0` worktree, pass local and GitHub review gates, then squash-merge the release PR. Revalidate the exact merged tree, build release assets from it, push an annotated tag, wait for the tag-triggered GHCR workflow, publish the GitHub release, and clean all release-only branch/worktree state.

**Tech Stack:** Rust/Cargo, Git worktrees, GitHub CLI and GitHub Actions, Docker/Buildx, GHCR, Trivy, Cargo Audit, Cargo Deny, ShellCheck, Bash packaging scripts.

## Global Constraints

- Target version and tag are exactly `0.8.0` and `v0.8.0`.
- All five workspace crates remain at their existing `0.8.0` versions; do not edit Cargo manifests or `Cargo.lock` for release preparation.
- Release preparation changes only `CHANGELOG.md`, `README.md`, and the approved documents under `docs/superpowers/`.
- Do not change Rust behavior, MCP schemas, auth scopes, inventory/token formats, service defaults, package behavior, or workflows.
- Do not publish workspace crates to crates.io.
- Do not enable ignored real-device tests or contact production devices.
- Publish exactly `rust-junosmcp_0.8.0_amd64.tar.gz` and `rust-junosmcp_0.8.0_amd64.tar.gz.sha256` as GitHub assets.
- Publish GHCR tags `0.8.0`, `0.8`, and `latest` through the existing tag-triggered `Release image` workflow.
- Do not publish the GitHub release until the tag-triggered GHCR workflow succeeds.
- Never move, recreate, or force-update `v0.8.0` after it is pushed. A source correction after tag publication becomes `v0.8.1`.
- Stop on any failed review, CI, security, packaging, checksum, tag-identity, or publication gate and rerun the complete affected gate after correction.

v0.8.0 combines Junos and SRX into one server and adds the operational controls needed to run that endpoint safely. Default builds expose the complete 26-tool Junos/SRX surface; Junos-only builds retain the 17-tool surface with `--no-default-features`.
