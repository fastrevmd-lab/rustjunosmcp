# v0.6.1 — SCP plumbing bug fixes (#56, #57)

**Date:** 2026-05-20
**Type:** Patch release (bug-fix only)
**Tracks:** fastrevmd-lab/RustJunosMCP#56, fastrevmd-lab/RustJunosMCP#57

## Goal

Fix two pre-existing bugs in `rust-junosmcp-core/src/tools/transfer_file.rs`
that the v0.6.0 `fetch_file` work inherited but did not introduce. Both surfaced
during the v0.6.0 two-stage opus review (SHOULD-4, SHOULD-5).

1. **#56** — `OpenSshScpRunner::{run,fetch}` await `child.wait()` before draining
   stdout/stderr pipes. If `scp` writes more than the kernel pipe-buffer
   capacity (~64 KiB on Linux) to stderr before exit, the child blocks on
   `write(2)` and `wait()` hangs until the MCP `timeout` cancels it.
2. **#57** — Host-key verification failures (exit 255, stderr contains
   `Host key verification failed.` or `REMOTE HOST IDENTIFICATION HAS CHANGED`)
   fall through to generic `JmcpError::ScpFailed`. Callers must substring-search
   scrubbed stderr to recognize them; remediation (refresh `known_hosts`) is
   distinct from a network timeout but the error surface doesn't say so.

## Non-goals

- No new MCP tool surface (tool count stays at **15**).
- No changes to `fetch_file` semantics — it inherits the fixes via shared
  scp plumbing.
- No rustez / rustnetconf version bumps.
- No refactor of the broader `transfer_file.rs` layout beyond extracting the
  shared scp-drive helper.

## Architecture

Both fixes are localized to `rust-junosmcp-core/src/tools/transfer_file.rs`
plus one additive variant on `rust-junosmcp-core/src/error.rs::JmcpError`. No
public API signature changes; no MCP tool argument or response changes.

### Fix #56 — concurrent scp child drive

Extract a shared helper:

```rust
async fn drive_scp_child(
    mut child: tokio::process::Child,
    ct: &CancellationToken,
    cancel_log_phase: &'static str,
) -> std::io::Result<ScpOutcome>
```

Body shape:

- Take `child.stdout` and `child.stderr` as piped readers.
- Outer `tokio::select!` preserves the cancellation arm exactly as it is today:
  on `ct.cancelled()`, log + `child.start_kill()` + `child.wait()` to reap, then
  return `io::Error::new(Interrupted, "cancelled")`.
- On the run arm, drive `wait`, `read_to_end(stdout)`, `read_to_end(stderr)`
  concurrently via `tokio::try_join!`. This eliminates the pipe-fill deadlock:
  even if scp emits >64 KiB of stderr before exit, the reader keeps the pipe
  draining so the child can finish writing and exit.
- Return `ScpOutcome { exit_code, stdout, stderr }` unchanged.

`OpenSshScpRunner::run()` and `OpenSshScpRunner::fetch()` shrink to:

1. Spawn the child via `Command::new("scp").args(&argv).kill_on_drop(true)...`.
2. Call `drive_scp_child(child, ct, "transfer_file.scp_diag" | "fetch_file.scp_diag")`.

The two functions differ only in which `argv` builder they call
(`build_scp_argv` vs `build_scp_fetch_argv`) and which log phase string they
pass; everything else dedupes.

### Fix #57 — structured `HostKeyMismatch` variant

Add to `JmcpError`:

```rust
#[error(
    "host key verification failed [code=host_key_mismatch]: router '{router}' was rejected; review or refresh the entry in {known_hosts_file}"
)]
HostKeyMismatch {
    router: String,
    known_hosts_file: PathBuf,
},
```

The exact Display string above matches the existing convention of sibling
variants (`ScpFailed`, `ConnectTimeout`) in `JmcpError`: a human-readable
lead clause, the `[code=...]` tag mid-string, then a colon-separated detail
clause. The `code=host_key_mismatch` tag remains the stable wire-format
discriminant; only the prose around it follows house style.

At the exit-255 dispatch in `transfer_file.rs` (currently lines ~1562–1577,
which converts non-timeout exit-255 outcomes into `JmcpError::ScpFailed`), add a
new branch *before* the generic `ScpFailed` fallback:

```rust
if outcome.exit_code == 255
    && (outcome.stderr.contains("Host key verification failed")
        || outcome.stderr.contains("REMOTE HOST IDENTIFICATION HAS CHANGED"))
{
    return Err(JmcpError::HostKeyMismatch {
        router: router_name.to_string(),
        known_hosts_file: cfg.known_hosts_file.clone(),
    });
}
```

Notes:

- The existing `Connection timed out` / `No route to host` → `ConnectTimeout`
  branch stays first; ordering doesn't matter since the substrings don't
  overlap, but keeping the network-timeout branch first preserves blame
  stability for any log scrapers.
- `cfg.known_hosts_file` is already in scope at the dispatch site
  (`TransferConfig` is the parameter to the surrounding function).
- The post-scp dispatch is duplicated today: once in `transfer_file.rs`
  (upload, lines ~1561–1579) and once in `fetch_file.rs` (download, lines
  ~183–194). To avoid hand-syncing two copies of the branch order, extract a
  shared classifier `classify_scp_failure(outcome, router_name, known_hosts_file) -> JmcpError`
  containing both the existing `ConnectTimeout` heuristic and the new
  `HostKeyMismatch` branch. Both call sites then collapse to a single
  `return Err(classify_scp_failure(...))` line when `outcome.exit_code != 0`.

## Error matrix after the change

| Scenario | Variant (after) | code= |
|---|---|---|
| scp exit 0 | `Ok(...)` | — |
| scp exit 255 + `Connection timed out` / `No route to host` | `ConnectTimeout` | `connect_timeout` |
| scp exit 255 + `Host key verification failed` / `REMOTE HOST IDENTIFICATION CHANGED` | **NEW** `HostKeyMismatch` | `host_key_mismatch` |
| scp exit 255 + other stderr | `ScpFailed` | `scp_failed` |
| scp exit ≠ 0, ≠ 255 | `ScpFailed` | `scp_failed` |
| cancellation token fired | `Cancelled` | `cancelled` |
| pre-existing `known_hosts` missing in strict mode | `KnownHostsMissing` | `known_hosts_missing` |

## Testing

### #56 — pipe-fill regression test

A standalone `tokio::test` that does not require scp on PATH:

- Spawn a tiny shell command via `tokio::process::Command::new("sh").args(["-c", ...])`
  that writes >128 KiB to stderr then exits 1, e.g.
  `head -c 131072 /dev/urandom | base64 1>&2; exit 1`.
- Wrap the resulting `Child` in `drive_scp_child` with a fresh
  `CancellationToken` and a `tokio::time::timeout(Duration::from_secs(5), ...)`.
- Assert the future returns within the timeout, exit_code == 1, stderr length
  ≥ 128 KiB.

Without the fix this test will hang and fail via the outer 5s timeout
(verifies the RED state); with the fix it completes in milliseconds.

### #57 — host-key mismatch variant tests

Unit tests in the `transfer_file` test module:

- Two synthesized `ScpOutcome` fixtures (exit 255, stderr = each of the two
  trigger substrings). Pass each through the dispatch helper and assert:
  - Returns `JmcpError::HostKeyMismatch` with the expected `router` + `known_hosts_file` fields.
  - `format!("{e}")` contains `[code=host_key_mismatch]` and the `known_hosts_file` path.
- One negative test: exit 255 + arbitrary stderr → still returns `ScpFailed`,
  not `HostKeyMismatch`.

### Existing tests

The stdio smoke test, the upload/download mock-runner tests, and the
strict-mode known_hosts tests must continue to pass unchanged. No tool surface
or argument changes; tool count stays at 15.

## Risks

- **Pre-existing callers pattern-matching on `ScpFailed`'s stderr substring.**
  None in this codebase as of v0.6.0 (verified via grep prior to design). External
  consumers of the library are unaffected because the MCP wire format flattens
  to `[code=...]` strings.
- **Helper extraction introduces a subtle behaviour change** if the cancellation
  arm of the new `tokio::select!` is structured differently than the originals.
  Mitigation: lift the cancellation arm verbatim into the helper, preserving the
  exact `tracing::info!` event names and the `start_kill` + `wait` reap order.
