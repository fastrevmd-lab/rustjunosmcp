# v0.8.0 Release Design

## Purpose

Publish the accumulated changes after `v0.7.0` as the production `v0.8.0`
release. The release includes the unified Junos/SRX server, HTTP resource
controls, Prometheus metrics, journald audit support, security fixes, container
SCP support, and the split-service upgrade migration.

The five surviving workspace crates already report version `0.8.0`. Release
preparation therefore changes release metadata and documentation, not package
versions or runtime behavior.

## Release Scope

The release preparation changes are limited to:

- promoting the current `CHANGELOG.md` `[Unreleased]` content to
  `[0.8.0] — 2026-07-16` and leaving a new empty `[Unreleased]` section;
- replacing the README's `v0.7.0` release banner with a `v0.8.0` summary and
  release-notes link;
- updating the README feature-scope section so `v0.8` is marked released and
  accurately describes the 26-tool default surface, 17-tool Junos-only
  surface, single-server deployment, and migration from the retired SRX
  service;
- adding this design and its implementation plan to the existing
  `docs/superpowers` history.

No Rust source, Cargo dependency, MCP schema, authorization scope, inventory
format, token format, service defaults, package installer behavior, or workflow
behavior changes as part of release preparation. Issue #110 remains open as
the accepted tracker for prerelease crypto dependencies inherited through
`russh`; it is disclosed in release notes but does not block `v0.8.0`.

## Release Path

All release-preparation changes are developed on branch `release/v0.8.0` in
the project-owned worktree `.worktrees/release-v0.8.0`. The branch is pushed
and opened as a pull request against `main`. The pull request must be ready for
review, have no unresolved review threads or requested changes, and pass every
repository CI and security check before merge.

After the release pull request is merged, local `main` is fast-forwarded to the
remote merge commit. The merged tree is revalidated before tagging. An
annotated `v0.8.0` tag is created on that exact merge commit with a concise
release annotation and pushed to `origin`.

The tag is immutable. It is never moved or force-updated after publication. A
source correction discovered after the tag is pushed becomes `v0.8.1`.

## Verification Gates

Before the release-preparation pull request is pushed:

- `cargo fmt --all --check` passes;
- `cargo clippy --workspace --all-targets -- -D warnings` passes;
- `cargo test --workspace --locked` passes;
- Junos-only and TLS-only builds/tests pass;
- the repository guard checks pass;
- Cargo audit and Cargo deny pass under the repository's accepted policy;
- Trivy reports no unignored vulnerability, secret, or production-image
  misconfiguration findings;
- ShellCheck passes for packaging scripts;
- the LXC archive builds and passes package plus Debian/Ubuntu distribution
  smoke tests;
- the container image passes healthcheck, non-root runtime, SCP upload, and SCP
  fetch smoke tests.

The host may run the exact commands behind `just` recipes when `just` is not
installed. Ignored real-device tests are not enabled without explicit lab
authorization, and no production device is contacted.

Before tagging, the same release/security/package gates are rerun against the
merged `main` tree. The tag may be pushed only when local `main`, `origin/main`,
and the verified commit are identical and the checkout is clean.

## Published Artifacts

The release publishes through the repository's existing distribution model:

1. Git tag `v0.8.0`.
2. GitHub release named `v0.8.0 — unified Junos/SRX server and operational
   controls`.
3. LXC archive `rust-junosmcp_0.8.0_amd64.tar.gz`.
4. Checksum file `rust-junosmcp_0.8.0_amd64.tar.gz.sha256` containing the
   archive's SHA-256 digest and filename.
5. GHCR image tags `ghcr.io/fastrevmd-lab/rust-junosmcp:0.8.0`, `:0.8`, and
   `:latest`, produced by the tag-triggered `Release image` workflow.

The GitHub release body summarizes the changelog, explicitly calls out the
breaking deployment change (the retired `rust-srxmcp` binary, systemd unit, and
port 30032), documents the one-release `JMCP_SRX_*` compatibility aliases, and
links the full comparison from `v0.7.0` to `v0.8.0`.

This release does not publish the workspace crates separately to crates.io.

## Publication Ordering

Publication follows this order:

1. Merge the release-preparation pull request.
2. Revalidate merged `main` and build/smoke-test the exact LXC artifact.
3. Create and push the annotated `v0.8.0` tag.
4. Monitor the tag-triggered GHCR workflow to a successful conclusion.
5. Reconfirm the LXC archive checksum and publish the GitHub release with both
   assets.
6. Verify the GitHub release, tag target, asset names/digests, successful image
   workflow, and remote image tags.
7. Remove the release worktree and local/remote release branch after the
   release is proven complete.

## Failure Handling

- Any pre-tag test, review, CI, security, packaging, or tree-identity failure
  stops the release. The failure is fixed on the release branch and the entire
  affected gate is rerun.
- A transient tag-workflow failure may be rerun after its cause is confirmed.
- A source-caused failure after `v0.8.0` is pushed does not permit moving the
  tag. Publication pauses and the correction is released as `v0.8.1`.
- GitHub release publication waits until the GHCR tag workflow succeeds, so a
  partially published release is not advertised as complete.
- Release assets are never uploaded from a dirty checkout or a commit that
  differs from the `v0.8.0` tag target.

## Completion Evidence

The release is complete only when all of the following are true:

- the release-preparation pull request is merged;
- `main` and `origin/main` point to the same clean commit;
- `v0.8.0` is an annotated tag pointing to that commit;
- the GitHub release is public and non-prerelease;
- the LXC archive and checksum assets exist and the checksum verifies locally;
- all three expected GHCR tags were produced by a successful workflow;
- the release worktree and both local and remote release branches are absent;
- real-device checks skipped by policy and the remaining #110 dependency risk
  are reported in the handoff.
